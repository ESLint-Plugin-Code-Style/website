# Impeccable — Design Context

This file is the source of truth for design direction on the `eslint-plugin-code-style` documentation website. It is read by every Impeccable skill (`critique`, `audit`, `shape`, `craft`, `polish`, etc.) before any design work.

Edit this file to push the direction further. Don't silently override it from a skill.

## Design Context

### Users

**Primary — The evaluating developer.** A React / TypeScript engineer choosing linting tooling for a new or existing project. They land on the site, read the hero, and within ~30 seconds decide whether this plugin belongs in their stack alongside ESLint and Prettier. They are skeptical (they've seen many linter plugins), literate (they know what a flat config is), and time-constrained (they are mid-task, not leisure-browsing). They want:

- A fast answer to *"why not just Prettier?"*
- A rule list they can scan to gauge philosophical fit
- An install snippet they can copy

**Secondary — The configuring developer.** Someone who has already installed the plugin and is looking up a specific rule, option, or preset config while writing `eslint.config.js`. They use the site as reference material, not marketing. Speed of lookup matters more than first-impression polish. Keyboard navigation, `⌘K` search, and direct URLs to rule pages should feel native.

**Tertiary — The curious passerby.** A developer linked from a blog post, Twitter, or GitHub. They may or may not become a user. The goal with this persona is not conversion — it is *memorability*. They should walk away thinking "this docs site has taste" and remember the brand.

### Brand Personality

**Three-word voice:** **Opinionated · Editorial · Handmade.**

- **Opinionated** — the plugin enforces style; the docs do too. No wishy-washy "you might consider." Confident assertions. Short sentences. Strong claims, backed by rule examples.
- **Editorial** — reads like a well-designed magazine or zine, not a product page. Strong typographic hierarchy. Generous whitespace. Long-form comfort. Every page earns its layout.
- **Handmade** — paper-cream background, SVG noise, handwritten annotations, zigzag dividers, rotated stickers. Feels *authored* — by a person with a point of view — not generated.

**Emotional goal for the first-time visitor:**
> "This has an author with taste. This isn't another shadcn-template docs site."

Secondary emotional goals: confidence (they trust the technical depth), delight (they notice small details — the crossed-out text, the handwritten correction, the squiggle draw-in), quiet pride in the craft (not loud — no confetti, no "✨ AI-powered ✨", no hero confetti animations).

### Aesthetic Direction

**The hybrid:** **Editorial format, linter content.**

- **Format = A (Zine / editorial).** Composition, type, rhythm, grid-breaking. Asymmetric columns. Pull-quotes. Margin notes. Handwritten section kickers instead of centered eyebrow + h2 patterns. Typographic indexes instead of card grids. Footnote-style asides instead of tooltips.
- **Content = B (Linter-as-metaphor).** Every visible example, animation, and ornament is about code being corrected. Hero's "wrong → fixed" motif propagates to every section. Stats crossed out and rewritten. Step numbers tinted with lint-semantic colors. Vignettes remain as live auto-fix gestures.
- **When A and B conflict, A wins.** Layout scales across the site; metaphor is best in small, curated doses.

#### Theme

Both light and dark modes, with equal design priority. Neither is the "default" — the user's `prefers-color-scheme` and the theme toggle decide.

- **Light mode:** paper-cream `#faf8f3` background + fractal-noise SVG grain at ~6% opacity + warm warm editorial accent. Feels like newsprint or a risograph zine.
- **Dark mode:** near-black editor `#0e1116` + same noise at ~5% opacity + same lint-semantic palette shifted cooler. Feels like a well-lit code editor at night.

#### Palette

- **Lint-semantic tokens:** `--lint-error` (warm red), `--lint-warn` (amber), `--lint-pass` (green), `--lint-info` (blue).
- **Secondary accent:** `--accent-violet` (`oklch(0.59 0.22 270)` family).
- Neutrals are tinted toward the violet hue, not pure gray.
- **No pure `#000` or pure `#fff`.** Always tinted.
- **No gradient text.** Solid colors for text.
- **No side-stripe borders greater than 1px.** Not on cards, not on list items, not on callouts.

#### Typography

**Three-font editorial pairing, locked:**

1. **Inter** — neutral running text, body prose, UI labels.
2. **JetBrains Mono** — code, data (numbers / counts / keys / tags), eyebrows, metadata.
3. **Caveat** — handwriting for voice: corrections, annotations, margin notes, section kickers. Used sparingly — rarity is the point.

**No fourth font.** If a design asks for a display serif, a slab, or a script, answer it with typographic contrast inside the existing three (weight, size, case, color, rotation) — not by adding a font.

**Scale rules:**
- Hero headings use fluid `clamp()`.
- App UI and docs navigation use fixed `rem` scale.
- Body line length 65–75ch.
- Line-height scales with line length; add 0.05–0.1 in dark mode to compensate for perceived-lighter type.

#### References (positive)

Study these for composition, type, and rhythm. Do not copy them:

- *Offscreen Magazine* — editorial zine pacing, handwritten marginalia
- *Works That Work* — typographic indexes, dense information composition
- *Kottke.org* / *Paul Ford's ftrain* — opinionated editorial voice, long-form prose composition
- *NYT feature spreads* — column breaks, asymmetric layouts, pull-quotes
- *Jack Cheng's blog* — restrained warmth, type-first
- Classic risograph zines — paper texture, limited palette, handmade feel

#### Anti-references (explicitly not this)

If the output resembles any of these, it's wrong:

- **shadcn / Tailwind template sites** — gray+zinc neutrals, `rounded-xl` everywhere, gradient heroes, icon-in-box feature cards. The default AI-SaaS look.
- **GitBook / generic docs themes** — tri-column layout, muted sidebar, forgettable prose.
- **Stripe / Linear clones** — dense micro-animations, gradient-glow accents, dark-mode-with-neon. Polished but over-imitated.
- **Corporate tech marketing** — safe, risk-averse, blue-brand, "Trusted by" logo wall.
- **AI-generated landing pages** — hero-metric template (big number + small uppercase label), rainbow gradient CTAs, identical card grids repeated four times, glassmorphism blur cards.

### Design Principles

These are the operating rules. Every skill's output must conform, or must earn the exception explicitly.

1. **The linter is the thesis.** Every visible example, number, and ornament shows code being corrected. If a section has no before→after gesture, it's probably wrong for this product. The hero's crossed-out-into-handwritten-correction motif is the ORGANIZING mechanic, not a one-off.

2. **Composition over decoration.** Distinctiveness comes from layout, type, and rhythm — not from adding more flourishes. Break the grid on purpose. Vary card shape per *section* (not per card). Strip before you amplify.

3. **Ornaments are rare by rule.** The rainbow top-strip, handwritten annotations, marker highlights, rotated stickers, zigzag dividers are *signatures*, not *patterns*. **Maximum 1–2 instances per page.** Repetition turns signature into wallpaper and kills the voice.

4. **Keyboard and screen-reader users are first-class.** WCAG 2.1 AA is the floor, not the ceiling. `:focus-visible` styles are mandatory; hover states must have focus parity; touch targets are 44×44 minimum; landmarks are thorough; `aria-current` / `aria-expanded` / `aria-hidden` are correctly applied. Never implement hover via JS `onMouseEnter`/`onMouseLeave` — use CSS states so keyboard focus inherits them.

5. **Editorial type discipline.** Line length 65–75ch on body prose. Fluid `clamp()` on hero / marketing headings. Fixed `rem` scale on docs and UI. Three fonts, no more. Monospace for data and code, handwriting for voice, Inter for running text. Light → dark theme adds 0.05–0.1 to line-height.

6. **No AI-template shortcuts.** If the shape appears by default in shadcn, Tailwind docs, or a typical AI-generated landing page, don't use it. In particular: no side-stripe borders >1px, no gradient text, no hero-metric templates, no icon-in-box feature cards, no identical card grids repeated across sections, no `rounded-xl` as a default, no dark-mode-with-glowing-accents.

7. **Respect prior invariants.** `src/data/rules.ts` is auto-generated — never edit. All strings live in `src/data/strings.ts`. All magic values live in `src/data/enums.ts`. Components are flat in `src/components/`, kebab-case filenames, single-export PascalCase. These are load-bearing conventions; redesign inside them, not around them.
